Design Walkthrough

Company Valuation Analytics Skill — Assignment 2

1. Why Company Valuation?

Domain Selection Rationale

Company valuation was chosen for several reasons that make it an ideal candidate for LLM skill authoring:

  1. Structured, repeatable methodology. Professional equity analysts follow the same three-method framework worldwide: Discounted Cash Flow (intrinsic value), Comparable Company Analysis (relative value), and Precedent Transactions (acquisition value). This makes the workflow precisely specifiable in a SKILL.md — every formula, every input, every output is deterministic.
  2. Multi-algorithm comparison is natural. The assignment requires running "at least two alternative approaches." Company valuation inherently compares three independent methods that approach the same question from different angles. Each produces a value range, and the cross-validation step (comparing all three) is a core part of real-world equity research — not an artificial requirement.
  3. Rich domain knowledge to encode. Valuation involves CAPM, WACC, terminal value, EV bridges, control premiums, industry benchmarks, and interpretation frameworks. This provides substantial material for REFERENCE.md without needing to fabricate domain context.
  4. Clear business output. The final deliverable — a buy/hold/sell recommendation backed by quantitative analysis — is immediately understandable to a business audience, making the report generation stage genuinely useful rather than decorative.
  5. Live data integration via MCP. Financial data is publicly available through APIs (Yahoo Finance), making MCP integration natural. The skill can fetch live data for any public company at runtime, demonstrating real-world applicability.

Domain Exclusions

We explicitly excluded RFM segmentation (used as the demo in the assignment brief). We also avoided pure ML prediction tasks (e.g., stock price prediction) because valuation is fundamentally about analysis and interpretation, not prediction — which better fits the assignment's emphasis on auditable, explainable workflows.

Key Design Decisions

Decision 1: JSON as inter-script data format. Each script reads JSON and writes JSON. This was chosen over CSV because financial data is hierarchical (target → records, peers → [records], transactions). JSON preserves this structure natively, eliminates column-ordering bugs, and makes intermediate outputs human-readable for debugging.

Decision 2: Separate feature engineering from modelling. The assignment emphasises "separate computation from interpretation." Going further, we also separated feature computation (Stage 2) from model execution (Stage 3). This lets the LLM inspect features before modelling — if WACC looks unreasonable, it can adjust config before committing to a full model run.

Decision 3: Two DCF terminal value methods. Instead of choosing between Gordon Growth and Exit Multiple, we compute both and blend them. This reduces sensitivity to any single assumption and gives the LLM a richer result to interpret. The sensitivity matrix covers 25 scenarios (5 WACC × 5 growth rate values).

Decision 4: Pluggable precedent transactions. Real M&A databases (Bloomberg, PitchBook) require paid access. We curated 33 real transactions (2020-2026) from SEC filings and press releases into data/transactions.json. The pipeline loads this file at runtime with priority: (1) user-provided file, (2) data/transactions.json, (3) hardcoded fallback. Users can expand the dataset without touching code.

Decision 6: Damodaran SOTA methodology. After initial results diverged from analyst consensus (e.g., MSFT showed as OVERVALUED when analysts said UNDERVALUED), we researched and adopted Damodaran's Jan 2026 parameters: US ERP = 4.46%, synthetic credit rating for cost of debt, total debt (not net debt) in WACC weights, revenue-driven FCF projections with median-of-4-CAGRs growth estimation, peer median exit multiple to avoid circularity, and IQR-based value ranges to eliminate outlier-driven absurd spreads.

Decision 5: Base64-embedded charts. Charts are encoded as base64 PNG images directly in the HTML rather than saved as separate files. This makes the report a single self-contained file — no broken image links when moving or sharing the report.

Error Handling Philosophy

The skill follows "fail early, fail clearly" — validation at Stage 1 catches data issues before expensive computation. Each error includes a fix field explaining exactly what to change. The pipeline uses exit codes (0 = success, 1 = failure) so downstream tools can detect failures programmatically.

Reproducibility

The synthetic data generator uses numpy.random.default_rng(seed=42) for reproducible output. The modelling script sets np.random.seed(42). Two runs with identical inputs and seeds produce byte-identical JSON outputs and visually identical reports.

4. What Failed and What Was Fixed

Challenge 1: Incomplete Fiscal Years from Yahoo Finance

Problem: When fetching 5 years of data via yfinance, the API often returns the oldest year with all-null financial statement fields (income statement columns exist but values are None). This produced 27 validation warnings on real AAPL data and yielded garbage ratio calculations (dividing by zero or treating null as 0).

Fix: Added an auto-filter in fetch_data.py that drops records where revenue is null. A log message reports how many incomplete years were dropped per company. Combined with a null guard in feature_engineering.py that skips records where both revenue and total_assets are null. After this fix: 0 warnings on AAPL, GOOGL, META, JNJ, PFE, MRK, LLY data.

Challenge 2: Balance Sheet Identity Mismatch

Problem: Yahoo Finance and synthetic generators sometimes produce balance sheet data where total_assets ≠ total_liabilities + total_equity due to rounding or minority interests. Early versions of the pipeline hard-failed on this.

Fix: Changed from strict equality to a 5% tolerance check. The validation script now computes |assets − (liabilities + equity)| / assets and only flags a WARNING if it exceeds 5%. This accommodates real-world data imprecision without silently accepting garbage data.

Challenge 2: Terminal Growth Rate ≥ WACC

Problem: The Gordon Growth Model formula TV = FCF × (1+g) / (WACC−g) produces division by zero or negative values when g ≥ WACC. This happened with certain config combinations (low risk-free rate + high growth assumption).

Fix: Added a guard in feature_engineering.py: if terminal growth ≥ WACC, automatically adjust terminal growth to WACC − 1% and print a warning. Also added a fallback in the DCF model: if the formula would divide by ≤ 0, use 20× terminal FCF as a simple cap.

Challenge 3: Negative FCF Breaking DCF

Problem: Companies with negative free cash flow produce meaningless or negative DCF valuations. Early versions didn't handle this, producing confusing results.

Fix: Added explicit negative FCF detection in run_models.py. When FCF < 0, the model still runs (producing whatever value the math gives) but appends a prominent warning. The cross-validation step correctly assigns LOW confidence when methods diverge significantly. The report template includes a warning banner in the DCF section.

Challenge 4: CCA Outlier Multiples (e.g., Amazon P/B = 51.86x)

Problem: Using raw min/max across all peer multiples produced absurd ranges (e.g., MSFT CCA range $135–$2,399 because Amazon's P/B of 51.86x applied to MSFT's book value). This made the report look unreliable.

Fix: Changed CCA and PT value ranges from raw min/max to 25th–75th percentile (IQR). This eliminates extreme outliers while preserving the full data for transparency. MSFT CCA range improved from $135–$2,399 to $216–$456.

Challenge 6: FCF CAGR Distorted by Capex Cycles

Problem: MSFT's FCF growth was only 3.2% despite 12% revenue growth, because AI data center capex ($64.5B) temporarily depressed free cash flow. Using FCF CAGR alone as the DCF growth driver produced a $166 fair value (vs $371 market price and analyst consensus of ~$430-$485).

Fix: Replaced single FCF CAGR with median of 4 CAGRs (revenue, earnings, EBITDA, FCF). This dampens capex-cycle distortions. Also switched to revenue-driven FCF projection (project revenue growth, apply FCF margin) instead of directly extrapolating FCF. MSFT growth rate improved from 3.2% to 12.15%, fair value from $166 to $372 (within $1 of market price).

Challenge 7: WACC Too High (Stale ERP)

Problem: Default ERP of 5.5% was arbitrary and too high vs Damodaran's Jan 2026 implied US ERP of 4.46%. Cost of debt used naive interest/debt ratio instead of market-based rates. WACC weights used net debt instead of total debt.

Fix: Adopted Damodaran methodology: ERP = 4.46%, risk-free rate = 4.37% (10Y Treasury), cost of debt via synthetic credit rating (interest coverage → rating → default spread), capital structure weights with total debt. MSFT WACC improved from 10.01% to 9.19%, matching Damodaran's Software sector WACC of 9.34%.

Challenge 5: Invalid Peer Tickers from API

Problem: When a user provides an invalid ticker symbol (e.g., "INVALIDXYZ"), yfinance raises an HTTP 404 error that crashed the entire fetch process, losing already-fetched valid data.

Fix: Wrapped each peer fetch in try/except that catches API errors and produces a "WARNING: skipping" message. After all fetches, the script re-checks that ≥3 valid peers remain. This is robust against typos and delisted tickers.

5. MCP Integration Architecture

Implemented MCP Server

scripts/mcp_server.py is a fully functional MCP server using JSON-RPC 2.0 over stdio (the standard transport for VS Code / Copilot skills). It exposes two tools:

  • fetch_financials — Fetches live financial data from Yahoo Finance for a target company and peers. Parameters: ticker, peers (comma-separated), years.
  • generate_synthetic — Generates synthetic data for testing. Parameters: target_ticker, sector, seed, num_companies, years.

The server was tested end-to-end:

  • initialize → returns protocol version 2024-11-05 and capabilities
  • tools/list → returns 2 tools with full JSON Schema input specs
  • tools/call with generate_synthetic → returns valid financial data

Data Flow (Two Equivalent Paths)

Path A (MCP): Agent calls fetch_financials tool → writes JSON to disk → CLI pipeline (Stage 1-6) Path B (CLI): Agent runs fetch_data.py → same JSON → same CLI pipeline (Stage 1-6)

Both paths produce identical JSON output; downstream scripts are agnostic to the data source.

Precedent Transactions (Pluggable)

M&A transaction data is stored in data/transactions.json (33 real deals, 2020-2026, from SEC filings and press releases, across 6 sectors: Technology 11, Healthcare 6, Energy 5, Consumer 4, Financials 4, Industrials 3). The pipeline loads this file at runtime with a priority chain: (1) user-provided path, (2) data/transactions.json, (3) hardcoded fallback in Python. Users can expand the dataset by editing the JSON file — no code changes needed.

6. Conclusion

The Company Valuation Analytics Skill demonstrates a complete, production-quality analytics pipeline with MCP integration that an LLM agent can orchestrate end-to-end. The three-method valuation framework (DCF, CCA, Precedent Transactions) provides natural multi-algorithm comparison. The 6-stage pipeline maps directly to the assignment's mandatory stages. Seven scripts (including an MCP server) are independently testable, reproducible (via seeds), and produce structured JSON for LLM inspection at each stage.

The skill was hardened iteratively against 5 real-world companies across 3 sectors (MSFT, AAPL, JNJ in US; MORN in Financials; NAUKRI.NS in India). Key hardening moments included: fixing incomplete Yahoo Finance data (auto-filtering), adopting Damodaran Jan 2026 WACC methodology, switching to revenue-driven FCF projections to handle capex cycles (MSFT fair value improved from $166 to $372 vs $371 market price), eliminating circular exit multiples, and applying IQR-based ranges to remove outlier-driven absurd spreads ($135–$2,399 → $216–$456 for MSFT CCA). The 33-transaction M&A database across 6 sectors enables meaningful Precedent Transaction analysis. All 7+ test scenarios passed, and the MCP server was tested end-to-end with initialize/tools-list/tools-call.
